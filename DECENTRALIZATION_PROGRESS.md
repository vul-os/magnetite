Added `KeyNameNaming`, a second real `Naming` provider behind `--features dmtap` (OFF by default): word-based, zero-authority key-names over an embedded 2048-word list (11 bits/word), plus optional `<words>@domain` display from a *local, non-authoritative* domain hint. **Honest arithmetic, stated plainly rather than papered over:** 8 words carry 88 bits and an Ed25519 key is 256, so no 8-word scheme — not ours, not DMTAP's — can losslessly encode a key. So two forms ship: `short_name` = 8 words = 88-bit domain-separated BLAKE3 fingerprint (the display name; resolves only for keys the node has *learned* out of band), and `full_name` = 24 words = 256 key bits + 8 checksum bits = 264 = 24x11 exactly (a pure encoding, exact round-trip for every key, BIP-39-*shaped* but not BIP-39). Wordlist derived mechanically from `/usr/share/dict/words` (Webster's Second "web2", **public domain**; no BIP-39 list vendored, so no Bitcoin-project licence question): keep `^[a-z]{4,7}$`, keep only the first word per 4-char prefix (every word unambiguous from its opening), sort/dedupe to 15491, take 2048 evenly spaced so names span the alphabet — and the list's invariants (2048, sorted, unique, unique 4-prefixes, ASCII) are asserted by tests so a corrupted list fails CI instead of silently renaming every key. Fail-closed throughout: unknown word, wrong word count, bad checksum, second `@`, or a domain that disagrees with the learned hint all return `None`/`Err`; a fuzz-ish junk battery (empty, control chars, astral chars, 200-word input) asserts no panic. **NOT NAMED `DmtapNaming`, deliberately:** `dmtap-core` is not on this machine and is not published, so its wordlist/packing/checksum/separator could not be read and compatibility could not be verified — names here will almost certainly NOT match DMTAP's for the same key, and putting an unverifiable compatibility claim in a *type name* is the worst place to put one. `docs/dmtap.md` says this in as many words and enumerates what awaits the real crate (`DmtapAuth`, wire-compatible names, `DmtapPubBlobStore`/MOTE, a DMTAP comms provider, and any DMTAP dependency at all — the `dmtap` feature adds none). Built that wire. **(1) TRANSPORT CHOICE — deliberately self-contained.** Plain blocking `std::net` TCP with 4-byte-length-prefixed JSON frames (64 MiB cap, announced-oversize refused rather than allocated) and an Ed25519 handshake over the node keypair from `magnetite_seams::identity`. **No DMTAP, no libp2p, no external service, no new dependency** — cross-node handoff is core game functionality and the architecture rests on DMTAP never being load-bearing, so putting the fleet on top of an optional protocol would invert that. Blocking sockets (not tokio) because `HandoffTransport::transfer` is a *sync* trait method; threads keep the seam signature untouched. A NAT-traversing alternative behind the same seam is mentioned in docs as possible-future and is **not** implemented. **(2) PEER AUTH — the address is not the identity.** Mutual challenge/response: client sends `Hello{pubkey, nonce}`, server answers `ServerHello{pubkey, nonce, sig}` signed over `tag||PROTO||both nonces||both pubkeys`, client verifies that signature **and that the presented key equals the key it pinned in the route**, then proves its own key the same way; server additionally checks the client key against an optional allowlist *before* spending a signature on it. Both nonces enter the transcript, and every later frame is signed over `transcript = BLAKE3(client_nonce||server_nonce)`, so a captured frame cannot be lifted into another session. Connecting to the right host:port is explicitly not accepted as proof of the right node — `PeerRoute` carries `{addr, pubkey}` and the pubkey is the load-bearing half. **(3) TWO-PHASE, EPOCH-FENCED HANDOFF — the correctness core.** Authority is the pair `(shard, epoch)`. `Offer{shard, epoch=e+1, state, hash}` → target verifies signature, re-hashes the bytes against the declared BLAKE3, checks `epoch > per-shard high-water`, and **stages** it (staged is explicitly *not* authoritative, TTL 60s) → signed `Ack` → source sends `Commit` → target promotes staged→owned and answers signed `CommitAck` → **only then** does the source release authority. **Every partial failure resolves to 'source still owns it'**: connect/handshake failure, target rejection, ack timeout, mid-transfer connection drop, and target crash all return `Err` with the source's `owned` entry and its state bytes untouched (the target's stage simply expires without ever becoming authoritative). The source never discards state before a verified `CommitAck`, so **'nobody owns the shard' is unreachable**. The one genuinely ambiguous window in any 2PC — commit delivered, commit-ack lost — is resolved by the **epoch fence** rather than hidden: the target holds `e+1`, the source holds the strictly lower `e`, and since every peer refuses any offer/commit at an epoch it has already passed, the stale owner cannot hand the shard on or resurrect itself; the higher epoch deterministically wins. The same monotonic check is what kills duplicates and replayed/stale handoffs, and the reserved epoch is burnt on failure so a retry cannot collide with an offer the peer may still have staged. **(4) DETERMINISM ACROSS MIGRATION** is asserted, not assumed: a control run ticks a deterministic toy world 1..=20 on one executor; the migrated run ticks 1..=10 on node A, ships the shard over a real socket, restores on node B and ticks 11..=20 — final snapshot hashes must be **byte-identical**, and the state hash is separately asserted continuous at the boundary (source snapshot == bytes received == post-restore snapshot). A second test proves a *failed* migration is invisible to the simulation. **(5) MULTI-NODE PLACEMENT:** `SpreadScheduler` places 8 shards across two real listeners keyed by their node pubkeys with capacity 6 vs 2 — the bigger box takes 6, the smaller 2 — and node A then ships B's share over the wire; both authority tables are then asserted disjoint (no shard owned twice, none owned by nobody).
