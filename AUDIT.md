## Backend Security

_The backend has several critical and high-severity security vulnerabilities concentrated in three areas: (1) the admin middleware is permanently broken — it unconditionally returns Forbidden for every caller, yet all admin routes are wired through auth_middleware (not admin_middleware), so the actual admin check is deferred to per-handler require_admin() DB lookups which do work, but an attacker who obtains any valid JWT still hits every admin route before the DB check fires; (2) the deposit handler accepts the USD credit amount directly from the user without cross-checking it against the Paystack-verified ZAR amount, enabling unlimited self-crediting on a single valid payment reference; (3) the webhook endpoint management router applies no auth middleware at all, so any unauthenticated caller can list, register, and delete webhook endpoints. Additional high-severity issues include: the games PUT /games/:id update handler has a full IDOR (no developer_id check), the refresh-token validation loads all live sessions and does a linear Argon2 scan (catastrophic DoS for any volume of sessions), the rate-limiter path matching uses /api/auth/ but the server mounts at /api/v1/auth/, making all per-route rate-limit rules permanently dead, and the Paystack webhook signature comparison is done with plain == rather than a constant-time compare, enabling timing attacks. The TOTP secret is stored in plaintext in the users table. Most SQL queries use parameterised binds correctly; the single exception is the dynamic ORDER BY clause in list_reviews which is built via format!() but the clause value is chosen from an internal whitelist match, not raw user input, so it is not injectable in practice._

## Frontend / Auth / Transport Security

_The frontend has a broad, consistent XSS-via-localStorage attack surface: every JWT is stored in localStorage under two different keys simultaneously ('token' and 'magnetite_token'), user objects are also persisted in plaintext, and none of the auth flows use HttpOnly cookies. The OAuth callback lacks any CSRF state-parameter validation on the client, and accepts a destination redirect from the query string without any origin restriction — an open redirect that could be trivially chained to steal the JWT token that lands in localStorage. The `?token=` URL fragment fallback in AuthCallback is unusual and creates a referrer/history leak path. No Content-Security-Policy header is set on either nginx config, meaning a single XSS sink exfiltrates all stored tokens. The known critical API-prefix mismatch (/api/... vs /api/v1/...) is confirmed: most REST calls 404 in production, but a mixed-prefix pattern partially exists where the Wise/payout endpoints explicitly use /api/v1/ while every other endpoint uses /api/. Payment data is console.logged in production. The VITE_PAYSTACK_PUBLIC_KEY is properly a public key so its exposure is expected, but the hardcoded user email placeholder ('user@example.com') passed to Paystack is a live bug. WebSocket authentication relies on the Bearer token being validated server-side on connection with no query-string exposure detected. No dangerouslySetInnerHTML was found. React renders user-supplied content (message text, bios, usernames, review text) safely via JSX text nodes — no stored XSS vector from content rendering._

## Missing Features vs Vision (DECISIONS.md §1/§1b/§4b + Economy)
