# Build & Distribution Pipeline

This document explains how a Rust game goes from source code to playable artifact
in the Magnetite marketplace.

---

## Overview

```
Developer repo (GitHub)
        │
        │  git push
        ▼
Magnetite GitHub App (webhook)
        │
        │  POST /api/v1/github/webhooks/github
        ▼
Backend — github.rs
  • Verify HMAC-SHA256 webhook signature
  • Record push event & commit SHA
  • Create / update build record in DB
        │
        │  (CI runs in your own GitHub Actions or provided workflow)
        ▼
Build workflow
  • cargo build --target wasm32-unknown-unknown --release
  • wasm-bindgen --target web --out-dir dist
  • Upload dist/ artifacts (WASM + JS bindings) to S3 / object store
  • POST build status back to /api/v1/github/repos/{owner}/{repo}/build-status
        │
        │  artifacts available
        ▼
Developer — submit game via Dashboard or API
  • POST /api/v1/games  (registers metadata + artifact URL)
  • PUT  /api/v1/developer/games/{id}/status  →  "pending_review"
        │
        │  admin review
        ▼
Admin — approve via Dashboard or API
  • PUT /api/v1/admin/games/{id}/approve
  • PUT /api/v1/admin/games/{id}/feature  (optional)
        │
        │  game goes live
        ▼
Marketplace — players discover, launch, play (browser WASM)
```

## GitHub App Integration

Magnetite ships a GitHub App that provides two things:

1. **Push webhooks** — every `push` event to a registered repository is received at
   `POST /api/v1/github/webhooks/github`. The backend verifies the `X-Hub-Signature-256`
   header using `HMAC-SHA256` with `GITHUB_WEBHOOK_SECRET`.

2. **Installation access tokens** — the backend generates short-lived JWTs (RS256,
   10-minute expiry) signed with `GITHUB_APP_PRIVATE_KEY` and exchanges them for
   GitHub installation tokens to call the GitHub Checks API and report build status.

### Register a repository

```
POST /api/v1/github/repos/register
Authorization: Bearer <access_token>
{
  "owner": "your-gh-user",
  "repo": "my-game",
  "installation_id": "12345678"
}
```

### Check build status

```
GET /api/v1/github/repos/{owner}/{repo}/build-status
Authorization: Bearer <access_token>
```

Response includes `commit_sha`, `status` (queued / in_progress / completed), and
`conclusion` (success / failure / cancelled).

## WASM build details

The `game-template/build.sh` script is the canonical build command:

```bash
#!/bin/bash
set -e
cargo build --target wasm32-unknown-unknown --release
wasm-bindgen \
  --target web \
  --out-dir ./dist \
  --out-name game \
  target/wasm32-unknown-unknown/release/<crate_name>.wasm
```

This produces:
- `dist/game_bg.wasm` — the compiled game module
- `dist/game.js` — JavaScript bindings generated by `wasm-bindgen`

The game entry point is the function annotated `#[wasm_bindgen(start)]`; the Bevy app
is launched from there via `App::new().add_plugins(…).run()`.

### Bevy + WASM notes

- Use `MinimalPlugins` (not `DefaultPlugins`) to avoid windowing/audio dependencies
  that do not compile to WASM.
- The `game-template` `Cargo.toml` declares `[lib] crate-type = ["cdylib", "rlib"]` — both
  are required for `wasm-bindgen` (cdylib) and Rust tests (rlib).
- `getrandom` must be enabled with the `js` feature to work in WASM:
  ```toml
  getrandom = { version = "0.2", features = ["js"] }
  ```

## Artifact storage

WASM artifacts (`_bg.wasm` + `.js`) are uploaded by CI to an S3-compatible object store.
The URL is recorded when a game is registered via `POST /api/v1/games`. The platform
serves this URL to browser clients; no re-upload through the API is required.

## Game lifecycle states

| State | Transitions to | Triggered by |
|-------|---------------|-------------|
| `draft` | `pending_review` | Developer calls `PUT /api/v1/developer/games/{id}/status` |
| `pending_review` | `approved` / `rejected` | Admin calls `PUT /api/v1/admin/games/{id}/review` |
| `approved` | `live` | Admin calls `PUT /api/v1/admin/games/{id}/approve` |
| `live` | `suspended` | Admin action |
| `suspended` | `live` | Admin action |

Only `live` games appear in the public marketplace (`GET /api/v1/games`).

## Developer earnings

Earnings are **receipt-backed and non-custodial**. Every sale is an atomic
wallet→wallet checkout that pays the developer directly and mints a signed
receipt; there is no accrual, no platform cut, and no payout step.

- `GET /api/v1/developer/earnings` — sums non-voided `payment_receipts`
  (`total − protocol_fee`) for the developer's games. `pending_payout` is
  always `0`: nothing is ever held on a developer's behalf.
- `GET /api/v1/developer/dashboard` — headline stats.
- `GET /api/v1/developer/games/:id/analytics` — per-game series.

A paid session additionally requires a **verified, non-voided receipt** before
it starts; it fails closed if one is missing.

`POST /api/v1/developer/payouts`, the 15/85 subscription split, and the minimum
payout threshold were **deleted**. See [Payments](../payments.md).

## CI workflow (reference)

A minimal `.github/workflows/game-ci.yml`:

```yaml
name: Build WASM
on: [push]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@stable
        with:
          targets: wasm32-unknown-unknown
      - run: cargo install wasm-bindgen-cli
      - run: bash build.sh
      - name: Upload artifacts
        uses: actions/upload-artifact@v4
        with:
          name: wasm-dist
          path: dist/
      - name: Report build status
        env:
          MAGNETITE_TOKEN: ${{ secrets.MAGNETITE_TOKEN }}
        run: |
          curl -X POST https://api.magnetite.gg/api/v1/github/repos/${{ github.repository_owner }}/${{ github.event.repository.name }}/build-status \
            -H "Authorization: Bearer $MAGNETITE_TOKEN" \
            -H "Content-Type: application/json" \
            -d '{"commit_sha":"${{ github.sha }}","status":"completed","conclusion":"success"}'
```
