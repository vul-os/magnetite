# Internationalisation (i18n)

Magnetite includes a lightweight i18n scaffold that provides:

- An English translation file (`src/i18n/en.json`) with all user-visible UI strings.
- A React context provider (`src/i18n/I18nProvider.jsx`) that loads locale dictionaries lazily.
- A `useTranslation()` hook (`src/i18n/useTranslation.js`) that resolves dot-separated key paths.

This is an **opt-in scaffold** тАФ existing pages are not rewired.  New pages and future migration work should adopt the `useTranslation` hook.

## Quick Start

```jsx
// Wrap the app in I18nProvider (optional тАФ add when ready):
import { I18nProvider } from './i18n/I18nProvider';

<I18nProvider>
  <App />
</I18nProvider>
```

In any component:

```jsx
import { useTranslation } from '../i18n/useTranslation';

function MyComponent() {
  const { t } = useTranslation();
  return <h1>{t('nav.home')}</h1>;  // "Home"
}
```

## Key Paths

Keys follow a `namespace.key` (or `namespace.subkey.leaf`) convention.  All current keys are defined in `src/i18n/en.json`.

Example paths:

| Key | Value (en) |
|-----|-----------|
| `common.loading` | LoadingтАж |
| `nav.marketplace` | Marketplace |
| `auth.errors.sessionExpired` | Your session has expired, please sign in again |
| `games.contentRatings.mature` | Mature (17+) |
| `wallet.transactionTypes.refund` | Refund |
| `social.blockUser` | Block user |
| `developer.developerEarnings` | Your earnings (70%) |
| `admin.refund` | Issue refund |

## Variable Interpolation

The `t()` function supports `{{variable}}` placeholders:

```js
// In en.json:
"greeting": "Hello, {{name}}!"

// In a component:
t('greeting', { name: 'Alice' })  // "Hello, Alice!"
```

If a variable is not provided, the placeholder is left as-is (`{{name}}`).

## Fallback Behaviour

1. The active locale dictionary is checked first.
2. If the key is not found, the English (`en`) dictionary is checked.
3. If the key is not found in English either, the key itself is returned.

This means the UI will never show an empty string тАФ worst case, it shows the key path (e.g., `auth.errors.weakPassword`).

## Adding a New Locale

1. Create `src/i18n/<locale>.json` тАФ copy `en.json` and translate the values.
2. Register the file in the `modules` map in `I18nProvider.jsx`:

```js
const modules = {
  'fr': () => import('./fr.json'),
  'de': () => import('./de.json'),
};
```

3. Add the locale code to the `detectLocale` supported list:

```js
detectLocale(['en', 'fr', 'de'])
```

## Testing

`src/i18n/useTranslation.test.js` covers:

- Resolution of English strings by key path.
- Fallback to the key string for missing keys.
- Variable interpolation.
- Graceful degradation when used outside `I18nProvider`.
- Locale switching via `setLocale`.
- Completeness spot-checks on required namespaces.

Run with:

```bash
npm test -- src/i18n/useTranslation.test.js
```

## Namespace Reference

The English file (`en.json`) organises strings into these namespaces:

| Namespace | Coverage |
|-----------|---------|
| `common` | Generic UI actions (save, cancel, loading, etc.) |
| `nav` | Navigation links |
| `auth` | Login, register, password reset, 2FA, errors |
| `games` | Game listing, reviews, content ratings |
| `wallet` | Balance, deposit, withdraw, transaction types |
| `social` | Friends, block/unblock, presence |
| `notifications` | Notification types and actions |
| `developer` | Dashboard, analytics, earnings, payouts |
| `admin` | Admin panel, moderation, refunds |
| `subscription` | Plans, upgrade, cancel |
| `errors` | Error pages (404, 500, 401, network) |
