# Internationalisation (i18n)

Magnetite includes a lightweight i18n scaffold that provides:

- An English translation file (`src/i18n/en.json`) with all user-visible UI
  strings.
- A React context provider (`src/i18n/I18nProvider.jsx`) that loads locale
  dictionaries lazily.
- A `useTranslation()` hook (`src/i18n/useTranslation.js`) that resolves
  dot-separated key paths with `{{variable}}` interpolation.
- A Vitest test suite (`src/i18n/useTranslation.test.js`) with 20+ cases
  covering key resolution, fallbacks, interpolation, and locale switching.

This is an **opt-in scaffold** — existing pages are not rewired. New pages and
future migration work should adopt the `useTranslation` hook. The
`NotificationPreferences` component is the first production component that uses
it end-to-end.

---

## Quick start

```jsx
// Optional — wrap the app in I18nProvider when ready:
import { I18nProvider } from './i18n/I18nProvider';

<I18nProvider>
  <App />
</I18nProvider>
```

In any component (works with or without `I18nProvider`):

```jsx
import { useTranslation } from '../i18n/useTranslation';

function MyComponent() {
  const { t } = useTranslation();
  return <h1>{t('nav.home')}</h1>;  // "Home"
}
```

When used without `I18nProvider`, the hook falls back to the bundled English
dictionary so the UI always shows real copy — never raw key strings.

---

## Variable interpolation

```js
// In en.json:
// "common": { "greeting": "Hello, {{name}}!" }

t('common.greeting', { name: 'Alice' })  // "Hello, Alice!"
```

Unknown placeholders are left as-is (`{{name}}`). Unknown keys return the key
string itself (e.g., `"auth.errors.unknown"`).

---

## Fallback chain

1. Active locale dictionary (from `I18nProvider` state).
2. English fallback dictionary (`en.json`, always bundled).
3. The key string itself.

The UI will never show an empty string.

---

## Adding a new locale

1. Create `src/i18n/<locale>.json` — copy `en.json` and translate the values.
2. Register the loader in `I18nProvider.jsx`:

   ```js
   const modules = {
     'fr': () => import('./fr.json'),
   };
   ```

3. Add the locale code to the `detectLocale` call:

   ```js
   detectLocale(['en', 'fr'])
   ```

`I18nProvider` detects the browser's preferred language via
`navigator.language` and loads the closest supported locale automatically.

---

## Locale detection

`detectLocale(supported)` in `I18nProvider.jsx` reads `navigator.language`,
strips the region tag (e.g., `"en-US"` → `"en"`), and returns the first
supported match. If no match is found, `"en"` is the default.

The `defaultLocale` prop on `I18nProvider` overrides detection (useful in
tests):

```jsx
<I18nProvider defaultLocale="en">…</I18nProvider>
```

## Testing

`src/i18n/useTranslation.test.js` (Vitest) covers:

- Resolution of English strings by key path (2-level, 3-level, 4-level).
- Missing key fallback to key string.
- Variable `{{…}}` interpolation.
- Graceful degradation outside `I18nProvider` (returns real English strings).
- `locale` and `setLocale` state.
- Completeness spot-checks for required namespaces.
- Specific checks for content ratings, wallet transaction types, developer
  analytics, admin moderation labels, and social block/unblock.

Run with:

```bash
npm test -- src/i18n/useTranslation.test.js
```

## Usage in production components

The only production component that uses `useTranslation` end-to-end as of this
wave is `NotificationPreferences` (`src/components/NotificationPreferences.jsx`).
It reads from the `notifPrefs` and `common` namespaces.

Existing pages use hardcoded English strings. Migration is opt-in; add
`useTranslation` calls when touching a page rather than doing a bulk sweep.
