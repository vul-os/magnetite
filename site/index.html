Anyone can re-run the match and check the result.

Clients send inputs, never state. A deterministic WASM sandbox steps the world. Every tick lands in a log that anybody can re-simulate from scratch.

magnetite is a Rust games platform whose one real asset is that its simulation is reproducible. The same ordered commands, the same seed, the same tick produce the same state on any host — so cheating is not something you detect by heuristics, it is something you prove from the record. Around that core sits a set of seams the platform programs against, each with a working offline default, so no provider is welded in. The project is currently being rebuilt around a no-central-cloud design; this page keeps what runs today apart from what is written down.

Honest status
The game core is genuine and tested: authoritative simulation, the Wasmtime sandbox, replay verification, the seam crate with offline defaults, and a node that measures its own hardware. The multi-node parts are real code proven only in-process and on a LAN — there is no NAT traversal, no relay and no WAN validation, so internet-scale clusters are undemonstrated. There is no on-chain payment rail: the shipped rail is a deterministic offline mock. Some of the pre-redesign central backend is still in the tree.

Seven seams, seven working defaults

Nothing in the game runtime, scheduler or payment path may name a provider-specific type — they see only these traits. Every seam ships a default that works with no network, no chain and no account, so the test suite runs fully offline and no external project is a hard dependency.

Seam 01

IdentityDefault ships

Identity is a keypair. Sign-a-challenge login over raw Ed25519, with the node able to mint short-lived audience- and scope-bound tokens so one keypair login can enter external comms systems.

Seam 02

NamingDefault ships

Human names are a display layer; the substrate is always raw keys. Short-hash addressing by default, with an optional word-based key-name provider that adds zero dependencies and exists to prove the seam really is swappable.

Seam 03

BlobStoreDefault ships

Content addressing: the hash is the id. A game needs no registry row to be identified. Local and HTTP stores today; peer-to-peer distribution is a later adapter behind the same trait.

Seam 04

DiscoveryDefault ships

A phonebook, never an authority. Nodes sign and lease their own advertisements; a tracker can refuse forgeries without gaining any say over who may host what. LAN plus a dumb HTTP tracker anyone can run, fanned out redundantly.

Seam 05

CommsProviderAdapters ship

Chat, voice, video and streaming are adapters, not a product we build. Matrix, Jitsi, LiveKit and Owncast sit behind one trait, with the old in-house stack demoted to a fallback.

Seam 06

PaymentRailMock only

Non-custodial by design: no balances, no payout queue, no custody. An entitlement is a signed receipt; hosting fees ride a channel; wagers settle from escrow.

What ships is the deterministic offline mock. No chain is wired up, so nothing here moves real money today.

Seam 07

InputProviderRead the caveat

Input is split into two classes that the compiler will not let you confuse. Deterministic input is replay-verifiable. Attested input — anything sensor-derived — is not, and never will be.

The default is a deterministic keyboard/gamepad queue that refuses attested events at runtime. A host-side gate screens attested events for rate, cooldown and human-reachable speed. No gesture producer exists in this repo — the seam is the socket, and today only the socket.

What's specified, not built

The redesign spec is deliberately candid about its own gaps, and so is this page. These are the things a reader could reasonably assume work, and which do not.

  • Internet-scale clustersCross-node shard handoff, cluster membership and player-follow redirects are real, tested code — but tested in-process and over a LAN. There is no NAT traversal, no relay and no WAN validation, so nodes must already be directly reachable. Federated compute between strangers' machines over the open internet is not demonstrated.
  • MoneyThe payment seam models atomic wallet-to-wallet splits, signed-receipt entitlements, hosting channels and wager escrow. What is implemented is a deterministic mock that signs receipts offline so CI can run. There is no chain integration, so no real payment has ever settled through it.
  • Trackerless discoveryDiscovery works over LAN and over ordinary HTTP trackers, fanned out so no single tracker is load-bearing. A DHT adapter is specified and unwritten.
  • Sensor input producersThe InputProvider seam accepts and screens attested events, but nothing in magnetite generates one. There is no camera capture, no pose model and no vendor SDK anywhere in the tree.
  • The redesign itselfmagnetite is mid-conversion from a conventional central-backend game platform to the no-cloud model described in DECENTRALIZATION.md. Fiat billing and custody are gone; parts of the old central backend are still present. Treat the spec as the intent and the status table above as the state.

Determinism is a proof, not a promise — and it has an edge

Most anti-cheat asks a server to guess whether a player is lying. Magnetite instead makes the match reproducible, so a third party can re-run it and compare. That is a much stronger claim, which is exactly why the places it does not apply have to be marked rather than glossed over.

Inputs, never state

A client can only ask. The host validates the request and steps the world itself, so a modified client cannot assert a position, a hit or a score — it can only send commands the host was always free to reject.

The same result anywhere

Game logic compiles to WASM and runs under a sandbox with a fuel budget, a memory cap and no access to the wall clock or OS randomness. Given the same state, ordered commands, tick and seed it produces the same output on any machine — which is what makes a log checkable at all.

Where it stops

Sensor input — a camera gesture, an IMU reading — cannot be re-derived from a log, because the pixels are gone and were never authoritative. A host can screen those events for human-reachable rate, cooldown and velocity, and that is genuinely all: a careful fake passes. The type system marks this class so nobody downstream mistakes it for proof.

Quick start

A game is a crate that implements one trait. The dev loop needs no server, no account and no network.

# install the CLI from the repo
cargo install --path magnetite-cli

# scaffold a crate implementing AuthoritativeGame
magnetite new my-game
cd my-game

# cargo build --release --target wasm32-wasip1
magnetite build

# sandboxed executor + live match, ZERO backend
magnetite dev

# bring a box: it measures its own hardware,
# content-addresses the game and advertises itself
magnetite node --advertise lan
  1. 1
    Write the game onceImplement AuthoritativeGame — a validate and a step, deterministic by construction. The SDK will not hand you a clock or a random number generator, because those are what break replay.
  2. 2
    Run it with no backendmagnetite dev compiles to wasm, boots the sandboxed executor and serves a playable match locally. No database, no account, no cloud — this path is the one most exercised.
  3. 3
    Bring your own boxmagnetite node measures cores and RAM, derives its own shard and player budget from them, and advertises what it can hold. Today that mesh is reachable over a LAN; see what's not built before planning an internet-wide fleet.
