## Backend Core APIs & Services

_The backend is genuinely multi-layered — auth (JWT sessions, real bcrypt, token rotation), wallet CRUD, DB-backed notifications/achievements/leaderboard/matchmaking queue/communities/streaming lifecycle, and the WebRTC voice signaling relay are all real and correctly wired. However, several critical revenue and payment paths are either completely mocked (PaymentService for Circle USDC: create_wallet, deposit_funds, withdraw_funds return fabricated random structs with no HTTP calls to the real Circle API) or half-baked (Paystack subscription sends to the real API but with a fake email). The email service is a total no-op (tracing::info! only, never instantiated). The payout pipeline has no actual money movement: process_single_payout() just flips a DB status column. Background scheduling for subscription renewals and payout batches is entirely absent from main.rs — only notification cleanup runs. Admin analytics_performance queries two non-existent tables (api_request_logs, websocket_connections) that have no migration, causing runtime failures. The Google OAuth callback decodes the JWT payload without verifying the RS256 signature, and Discord/GitHub/GitLab callbacks skip state/CSRF validation. The services/games.rs service layer is five todo!() stubs. The games POST /games handler hardcodes developer_id as the nil UUID. The payout fee split (30/70 in code) contradicts the declared 15/85 in DECISIONS.md. Streaming and voice at scale depend on undeployed external infrastructure (MediaMTX, SFU) that is correctly documented as the scale path but means no video/voice actually flows without separate ops work._

## Backend Integrations & Infrastructure

_The backend infrastructure is a mixed bag of genuinely working plumbing, honest stubs, and a few dangerous mocks. The critical payment path — Circle USDC wallet creation/balance/transfer, Paystack payment verification, and developer payouts — is entirely simulated: PaymentService's wallet and payout methods return fabricated data with random addresses and always-succeed statuses, PaymentService is never instantiated in any request handler (only in tests), and no real money ever moves. Email is similarly hollow: the lettre crate is declared in Cargo.toml but EmailService.send_email() only logs "Email would be sent…" with no transport. Verification tokens can be generated in the DB but are never dispatched to users, so email confirmation and password reset are non-functional. The WASM build pipeline is a record-keeping stub: trigger_wasm_build() writes a DB log row but never launches wasm-pack or any subprocess. GitHub App JWT auth and webhook signature verification are real and correct, but depend on env vars that panic on startup if absent. The distribution layer (version/artifact CRUD, play manifest) is real SQL; the play manifest endpoint works correctly but artifact_url stays NULL because no build worker ever reports back. Redis rate limiting is real but falls back silently to in-memory if Redis is absent. The session cleanup job and backup job are well-written but are never scheduled (not tokio::spawn'd in main.rs). Anti-cheat detection functions are genuine heuristics but are never called from any game-session handler — flags arrive only via a raw SQL INSERT in the game webhook, bypassing the service entirely. Five core services/games.rs functions (get_all_games, create_game, etc.) are literal todo!() panics. The new comms/voice/streaming layer from Waves 6-9 is the most complete new work: real WS broadcast for chat+presence, real DB lifecycle for streams, and an honest proxy/redirect for HLS with MediaMTX documented as a deploy dependency._

## Backend Realtime + Comms + Economy

_The backend realtime and comms layer has genuine substance: ws/comms.rs (chat/presence) and ws/voice.rs (WebRTC SDP/ICE signaling) are real, wired, and functional against a live DB — they handle auth, DB persistence, and in-memory fan-out correctly. The points economy service (award, spend, history, leaderboard, season reset) is similarly real with proper transactional semantics and DB locking. The marketplace purchase flow at the DB level is correct — it moves balances and creates entitlements in a transaction. The streaming lifecycle API (create, go-live, stop, viewer-count) is real metadata management backed by real SQL. However, critical capability gaps permeate the layer. The game WebSocket loop (ws/game.rs) is entirely unmounted — the file exists with dead_code suppressed but GameWsHandler::router() is never merged into the Axum app, so no multiplayer game session can be established. The revenue-share arithmetic in both marketplace.rs and payout.rs is demonstrably broken (0.7% instead of 70% — confirmed by running cargo test, two tests FAIL). Payments are not real: wallet deposits trust a caller-supplied payment_id string without contacting Circle or Paystack, withdrawals write 'pending' to DB with no processor, and subscriptions hardcode payment_provider='stripe' in a completed record without any API call — the real SubscriptionService (services/payment.rs) is marked 'not yet wired'. Developer payouts are similarly pending-only with no disbursement. The WASM build pipeline writes log rows but does not invoke cargo. The award_points and season_reset endpoints have no admin restriction despite code comments saying they need one. The streaming watch path is infrastructure-contingent: HLS delivery and RTMP egress both require a separately deployed MediaMTX server, which is documented but not provided; the backend has no media-server webhook receiver to auto-update hls_url. In summary: chat/voice signaling/points economy are real foundations; payments/payouts/WASM builds/game loop are either stubs or critically broken._

## Frontend: src/pages/**, src/hooks/**, src/context/**, src/api/client.js

_The frontend presents a polished visual shell built on top of a pervasive mock-data infrastructure. Roughly 80% of the data visible to users is fabricated and will remain so in production. The most critical gap is useWebSocket.js (useWebSocket.js:42-98), which unconditionally substitutes a silent createMockSocket for any real WebSocket connection — this means every real-time feature (comms, lobby, matchmaking, spectator) is permanently deaf to the backend. Auth has a silent mock-JWT fallback so any login "works" even with no backend. Every money-related flow (deposit, withdraw, subscription, earnings payout) either silently succeeds with optimistic state or logs to console instead of calling the API. All admin CRUD actions (ban, approve, process payout, save settings) are fake setTimeout stubs with no network calls. The three developer-workflow pages (GameStudio, GameDeploy, DeploymentStatus) simulate GitHub OAuth and CI builds with local timeouts and Math.random() progress bars. Streaming is documented-only at the media layer: HLS.js is not bundled, RTMP relay is a form field with no relay infrastructure, and the video player renders a placeholder for all streams. The comms hooks (communities, channels, messages, voice) have real API plumbing that will work if the backend answers, but all initialise from hardcoded constants and silently swallow errors, so the UI always appears populated regardless of backend health. In total: the codebase is a convincing demo that would require significant backend connectivity and removal of mock fallbacks before it could be called a real product._

## SDK + Game Templates + Tests/Docs

_The magnetite-sdk platform clients (comms, points, marketplace, cloud-save, streaming) are well-designed, fully-tested I/O-free type stubs: they build correct messages and maintain local state, but provide zero transport. A game integrating them must supply its own WebSocket send/receive loop. The game templates (game-template, fps, motorsport) implement real, deterministic server-authoritative game logic with genuine Rust unit tests (8–38 per crate), and the base game-template has a real WASM CI build pipeline. The fps template's gamepad button path is an explicit stub that always returns false; the HUD is a dead-code placeholder. The WASM build CI (game-ci.yml) only covers the arcade template — fps and motorsport have no CI coverage at all. The 74 backend Rust tests cover only serialisation, JWT, and HTTP status codes; there are zero integration tests against a real Axum router or database. Multiple backend services exist but are entirely unwired: email (logs only), verification, achievements, leaderboard, tournaments, anti-cheat, and matchmaking server allocation. The Circle and Paystack payment services have real HTTP client code but are marked `not yet wired` and not mounted in the router. Frontend hooks for points, marketplace, and communities initialise from hardcoded mock constants and silently fall back on API error, meaning the UI renders mock data even in production if the backend is unavailable. All four e2e specs explicitly assert on mock data rather than live API responses. Streaming HLS watch and RTMP egress require an external MediaMTX deployment that is not shipped; the StreamPlayer shows a placeholder when no HLS URL is present._
